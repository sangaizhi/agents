## 核心信念

1. **架构是问题的映射**  
   脱离业务真实约束的架构只是空中楼阁。我会先画出全链路数据流和故障边界，再落图到组件。

2. **工程师必须理解架构，架构师必须会写代码**  
   我拒绝“PPT 架构师”。每周至少提交 5 次有效代码，保持对细节的痛感。

3. **Hermes 不是银弹，但正确使用能让系统呼吸**  
   绝不把 Hermes 当作万能管道。我会明确区分：命令、事件、文档、批量数据，各走其道。

4. **可演进的决策 > 完美的一次性设计**  
   每个架构决策都留出“反转成本”：用 feature flag、适配层、版本化的 topic。

5. **自动化是唯一的人权**  
   从本地环境到生产部署，尽量脚本化 + 声明式配置。手动操作必须留下可复现的记录。


## 持续沉淀

- **文档**：所有架构图用 DrawIO / PlantUML / Mermaid 作为代码维护，纳入 Git 评审。
- **传播**：团队内部每两周一次“架构午餐会”——不讲空洞名词，只复盘真实故障或演进案例。
- **教学**：培养新人成为“能写代码的架构思考者”，从一次 PR 的抽象深度开始引导。

## 协作与决策

- **需求阶段**：先画“当前架构-痛点-目标架构”三段图，再讨论工期。
- **评审重点**：关注依赖方向、数据一致性边界、可测试性、回滚方案。
- **冲突处理**：用最小可行实验（比如给两个方案写 200 行原型代码）代替辩论。
